# Wiki Homepage Schema

**File**: `Home.md`
**Purpose**: Main entry point for GitHub Wiki documentation portal

## Structure

```markdown
# [Project Title]

[Welcome/introduction paragraph]

## Features

| # | Feature | Status | Description | Quick Start |
|---|---------|--------|-------------|-------------|
| [3-digit number] | [Link to feature page] | [Status emoji + text] | [One-line description] | [Link to quickstart section if available] |
| ... | ... | ... | ... | ... |

- **Quick Start**: [Description]

- **Data Model**: [Description]

## Field Specifications

### Project Title
- **Type**: String
- **Example**: "Chocolandia Kube Documentation"
- **Required**: Yes

### Welcome Paragraph
- **Type**: Markdown text
- **Max Length**: 500 characters
- **Example**: "Welcome to the chocolandia_kube homelab documentation! This Wiki provides comprehensive guides for all infrastructure features."
- **Required**: Yes

### Feature Table

#### Column: # (Feature Number)
- **Type**: String (3 digits with leading zeros)
- **Pattern**: `[0-9]{3}`
- **Example**: "001", "010", "025"
- **Required**: Yes

#### Column: Feature
- **Type**: Wiki link to feature page
- **Pattern**: `[Feature Name](Feature-{number}-{name})`
- **Example**: `[K3s Cluster Setup](Feature-001-K3s-Cluster-Setup)`
- **Required**: Yes

#### Column: Status
- **Type**: Enum with emoji prefix
- **Values**:
  - `‚úÖ Complete` - Feature has tasks.md and is implemented
  - `üöß In Progress` - Feature has plan.md but not complete
  - `üìù Planning` - Feature only has spec.md
- **Required**: Yes

#### Column: Description
- **Type**: Plain text (no markdown formatting in table cell)
- **Max Length**: 100 characters
- **Example**: "DNS ad blocker deployment via Kubernetes manifests"
- **Required**: Yes
- **Extraction**: From spec.md - first paragraph after title

#### Column: Quick Start
- **Type**: Wiki section link or "‚Äî" if unavailable
- **Pattern**: `[Quick Start](Feature-{number}-{name}#quick-start)` or `‚Äî`
- **Example**: `[Quick Start](Feature-003-Pihole#quick-start)`
- **Required**: No (depends on quickstart.md existence)

### Documentation Types Section
- **Type**: Markdown list with definitions
- **Required**: Yes
- **Purpose**: Help readers understand what each documentation type contains

### Contributing Section
- **Type**: Ordered list + warning text
- **Required**: Yes
- **Purpose**: Explain how to update documentation (edit in repo, not Wiki)

### Footer
- **Type**: Markdown italic text with timestamp and link
- **Pattern**: `*Last updated: {ISO 8601} | Generated by [script name](script URL)*`
- **Example**: `*Last updated: 2025-11-13T10:30:00Z | Generated by [sync-to-wiki.sh](https://github.com/.../sync-to-wiki.sh)*`
- **Required**: Yes

## Validation Rules

1. **Feature table rows**: MUST be sorted by feature number (ascending)
2. **Feature numbers**: MUST be unique (no duplicates)
3. **Wiki links**: MUST reference existing Wiki pages (validated during sync)
4. **Status values**: MUST be one of the three allowed values
5. **Timestamp**: MUST be valid ISO 8601 format
6. **Markdown syntax**: MUST be valid GFM (GitHub Flavored Markdown)

## Documentation Types

Each feature includes:

- **Quick Start**: Step-by-step deployment guide for getting the feature running quickly
- **Specification**: User scenarios, functional requirements, and success criteria (what users need)
- **Implementation Plan**: Technical context, architecture decisions, and constitution compliance check
- **Data Model**: Entities, relationships, and state transitions for features involving data
- **Research**: Technology decisions, alternatives considered, and rationale
- **Tasks**: Detailed implementation task breakdown with priorities and dependencies

## Contributing

Documentation is maintained in the [chocolandia_kube repository](https://github.com/cbenitezpy-ueno/chocolandia_kube).

To update documentation:
1. Edit files in `specs/XXX-feature-name/` directory
2. Run `./scripts/wiki/sync-to-wiki.sh` to sync to Wiki
3. Or wait for automated sync (runs on every merge to main - future enhancement)

**Do not edit Wiki pages directly** - changes will be overwritten on next sync.

```

## Error Handling

| Error Condition | Handling |
|-----------------|----------|
| No features found in specs/ | Error: "No features to sync" - abort |
| Duplicate feature numbers | Error: "Duplicate feature {number}" - abort |
| Invalid status value | Warning: Default to "üöß In Progress" |
| Missing spec.md for feature | Warning: Skip feature, log to stderr |
| Broken Wiki link | Error: "Link validation failed" - abort sync |
